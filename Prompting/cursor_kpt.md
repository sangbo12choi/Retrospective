```markdown
# KPT 회고 가이드

## KPT 회고란?

KPT 회고는 **Keep, Problem, Try**의 약자로, 팀이나 개인이 프로젝트나 일정 기간 동안의 활동을 돌아보고 개선점을 도출하는 회고 방법론입니다.

## 구성 요소

### 📌 Keep (계속 유지할 것들)
- **목적**: 잘 되고 있는 것, 계속 유지해야 할 좋은 점들을 식별
- **예시**:
  - "매일 스탠드업 미팅이 효과적이었음"
  - "코드 리뷰 프로세스가 품질 향상에 도움이 됨"
  - "문서화를 꾸준히 유지한 것이 좋았음"

### ⚠️ Problem (문제점)
- **목적**: 개선이 필요한 문제점이나 아쉬운 점들을 명확히 파악
- **예시**:
  - "커뮤니케이션 부족으로 인한 작업 중복 발생"
  - "배포 프로세스가 복잡하고 시간이 오래 걸림"
  - "테스트 커버리지가 낮음"

### 🚀 Try (시도해볼 것들)
- **목적**: Problem을 해결하기 위한 구체적인 개선 방안 제시
- **예시**:
  - "슬랙 채널에 일일 진행 상황 공유하기"
  - "CI/CD 파이프라인 자동화 도입 검토"
  - "단위 테스트 작성 가이드라인 수립"

## 회고 진행 방법

### 1단계: 준비
- 회고 날짜와 참석자 확인
- 회고 기간과 범위 설정
- 회고 장소/도구 준비 (온라인/오프라인)

### 2단계: Keep 작성
- 각자 또는 팀원들과 함께 잘 되었던 점들을 나열
- 긍정적인 피드백을 중심으로 작성
- 구체적인 사례나 데이터가 있으면 더 좋음

### 3단계: Problem 작성
- 문제점을 객관적으로 기록
- 비난이 아닌 개선점으로 접근
- 원인 분석보다는 현상 중심으로 작성

### 4단계: Try 작성
- Problem과 연결된 구체적인 해결 방안 제시
- 실행 가능한 수준으로 구체화
- 우선순위를 고려하여 작성

### 5단계: Action Items 수립
- Try 항목들을 실행 가능한 작업으로 변환
- 담당자와 기한을 명확히 지정
- 다음 회고에서 검증할 수 있도록 기록

## 팁

✅ **DO (권장사항)**
- 구체적이고 측정 가능한 내용으로 작성
- 긍정적인 분위기 유지
- 모든 의견을 존중하고 수용
- 실행 가능한 Try 항목 작성

❌ **DON'T (피해야 할 것)**
- 개인 비난이나 지적
- 추상적이고 모호한 표현
- Try 항목을 너무 많게 작성 (3-5개 권장)
- 회고 후 실행 계획 없이 끝내기

## 템플릿 사용법

1. `KPT_TEMPLATE.md` 파일을 복사하여 새로운 회고 파일 생성
   - 예: `KPT_2024_01_15.md`
2. 각 섹션을 채워나가며 회고 진행
3. 회고 후 Action Items를 추적 관리
4. 다음 회고에서 이전 Try 항목의 실행 결과를 확인

## 예시

`KPT_EXAMPLE.md` 파일을 참고하여 실제 회고 예시를 확인하세요.
```

```markdown
## 회고 날짜

- 프로젝트/기간: 웹 애플리케이션 개발 (2024-01-01 ~ 2024-01-15)

---

## 📌 Keep (계속 유지할 것들)
> 잘 되고 있는 것, 계속 유지해야 할 좋은 점들을 기록합니다.

- **일일 스탠드업 미팅**: 매일 오전 10시 15분 스탠드업이 팀원 간 정보 공유에 매우 효과적이었음
- **코드 리뷰 프로세스**: 모든 PR에 대해 최소 1명의 리뷰를 받는 프로세스가 버그 발견에 큰 도움이 됨
- **문서화 습관**: 주요 기능 개발 시 README와 API 문서를 함께 작성한 것이 좋았음
- **Git 브랜치 전략**: feature/기능명 형식의 브랜치 네이밍이 명확하고 관리하기 쉬웠음

---

## ⚠️ Problem (문제점)
> 개선이 필요한 문제점이나 아쉬운 점들을 기록합니다.

- **커뮤니케이션 부족**: 백엔드와 프론트엔드 간 API 스펙 변경 시 즉시 공유하지 않아 작업 중복 발생
- **배포 프로세스**: 수동 배포로 인해 배포 시간이 오래 걸리고 실수 가능성 존재
- **테스트 커버리지**: 단위 테스트 작성이 일관되지 않아 일부 모듈은 테스트가 부족함
- **문서 접근성**: 문서가 여러 곳에 분산되어 있어 필요한 정보를 찾기 어려움

---

## 🚀 Try (시도해볼 것들)
> 다음에 시도해볼 개선 방안이나 새로운 접근 방법을 기록합니다.

- **API 변경 알림 시스템**: API 스펙 변경 시 슬랙 채널에 자동 알림을 보내는 시스템 구축
- **CI/CD 파이프라인 도입**: GitHub Actions를 활용한 자동 배포 프로세스 구축 검토
- **테스트 가이드라인 수립**: 단위 테스트 작성 가이드라인 문서화 및 코드 리뷰 시 체크리스트 추가
- **문서 중앙화**: 모든 프로젝트 문서를 Notion 또는 Confluence에 통합 관리

---

## 📝 Action Items (실행 계획)
> Try 항목들을 구체적인 실행 계획으로 변환합니다.

| 항목 | 담당자 | 기한 | 상태 |
|------|--------|------|------|
| API 변경 알림 시스템 구축 | 김개발 | 2024-01-22 | 진행중 |
| CI/CD 파이프라인 설계 및 검토 | 박기획, 김개발 | 2024-01-25 | 예정 |
| 테스트 가이드라인 문서 작성 | 이디자인, 김개발 | 2024-01-20 | 예정 |
| 문서 중앙화 작업 | 박기획 | 2024-01-18 | 완료 |

---

## 💭 추가 메모
> 기타 기록하고 싶은 내용

- 다음 스프린트에서는 더 많은 사용자 피드백을 수집할 수 있도록 베타 테스터 그룹 구성 검토 필요
- 성능 모니터링 도구 도입을 다음 분기 계획에 포함시키기로 함
```

## 생성된 파일

1. **`KPT_TEMPLATE.md`** - 회고용 템플릿
2. **`README.md`** - KPT 회고 가이드
3. **`KPT_EXAMPLE.md`** - 실제 사용 예시

## KPT 회고 진행 방법

### 1단계: 템플릿 복사
`KPT_TEMPLATE.md`를 복사해 날짜별 회고 파일을 만드세요.
- 예: `KPT_2024_01_15.md`

### 2단계: 각 섹션 작성
- **Keep**: 잘 되고 있는 점, 계속 유지할 것
- **Problem**: 개선이 필요한 문제점
- **Try**: 시도해볼 개선 방안

### 3단계: 실행 계획 수립
Try 항목을 Action Items로 변환하고 담당자와 기한을 지정하세요.

### 4단계: 추적 및 반영
다음 회고에서 이전 Try 항목의 실행 결과를 확인하고 반영하세요.

## 핵심 원칙

- 구체적이고 측정 가능하게 작성
- 긍정적인 분위기 유지
- 실행 가능한 Try 항목 작성 (3-5개 권장)
- Action Items를 명확히 지정하고 추적

`README.md`에 상세 가이드가, `KPT_EXAMPLE.md`에 예시가 있습니다. 필요하면 추가 도움을 드리겠습니다.
