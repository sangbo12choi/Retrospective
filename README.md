# KPT 회고 가이드

## KPT 회고란?

KPT 회고는 **Keep, Problem, Try**의 약자로, 팀이나 개인이 프로젝트나 일정 기간 동안의 활동을 돌아보고 개선점을 도출하는 회고 방법론입니다.

## 구성 요소

### 📌 Keep (계속 유지할 것들)
- **목적**: 잘 되고 있는 것, 계속 유지해야 할 좋은 점들을 식별
- **예시**:
  - "매일 스탠드업 미팅이 효과적이었음"
  - "코드 리뷰 프로세스가 품질 향상에 도움이 됨"
  - "문서화를 꾸준히 유지한 것이 좋았음"

### ⚠️ Problem (문제점)
- **목적**: 개선이 필요한 문제점이나 아쉬운 점들을 명확히 파악
- **예시**:
  - "커뮤니케이션 부족으로 인한 작업 중복 발생"
  - "배포 프로세스가 복잡하고 시간이 오래 걸림"
  - "테스트 커버리지가 낮음"

### 🚀 Try (시도해볼 것들)
- **목적**: Problem을 해결하기 위한 구체적인 개선 방안 제시
- **예시**:
  - "슬랙 채널에 일일 진행 상황 공유하기"
  - "CI/CD 파이프라인 자동화 도입 검토"
  - "단위 테스트 작성 가이드라인 수립"

## 회고 진행 방법

### 1단계: 준비
- 회고 날짜와 참석자 확인
- 회고 기간과 범위 설정
- 회고 장소/도구 준비 (온라인/오프라인)

### 2단계: Keep 작성
- 각자 또는 팀원들과 함께 잘 되었던 점들을 나열
- 긍정적인 피드백을 중심으로 작성
- 구체적인 사례나 데이터가 있으면 더 좋음

### 3단계: Problem 작성
- 문제점을 객관적으로 기록
- 비난이 아닌 개선점으로 접근
- 원인 분석보다는 현상 중심으로 작성

### 4단계: Try 작성
- Problem과 연결된 구체적인 해결 방안 제시
- 실행 가능한 수준으로 구체화
- 우선순위를 고려하여 작성

### 5단계: Action Items 수립
- Try 항목들을 실행 가능한 작업으로 변환
- 담당자와 기한을 명확히 지정
- 다음 회고에서 검증할 수 있도록 기록

## 팁

✅ **DO (권장사항)**
- 구체적이고 측정 가능한 내용으로 작성
- 긍정적인 분위기 유지
- 모든 의견을 존중하고 수용
- 실행 가능한 Try 항목 작성

❌ **DON'T (피해야 할 것)**
- 개인 비난이나 지적
- 추상적이고 모호한 표현
- Try 항목을 너무 많게 작성 (3-5개 권장)
- 회고 후 실행 계획 없이 끝내기

## 템플릿 사용법

1. `KPT_TEMPLATE.md` 파일을 복사하여 새로운 회고 파일 생성
   - 예: `KPT_2024_01_15.md`
2. 각 섹션을 채워나가며 회고 진행
3. 회고 후 Action Items를 추적 관리
4. 다음 회고에서 이전 Try 항목의 실행 결과를 확인
